# torch-env

Shared conda env for torch-based research repos. One env, two platforms:

| Platform     | Spec                       | Lockfile                          | Accelerator |
| ------------ | -------------------------- | --------------------------------- | ----------- |
| Linux x86_64 | `environment.linux.yaml`   | `lockfiles/requirements.linux.txt` | CUDA 13     |
| macOS arm64  | `environment.mac.yaml`     | `lockfiles/requirements.mac.txt`   | MPS         |

Conda installs only python, pip, and a consistent BLAS/OpenMP stack.
Everything else (including torch) comes from pip so we always pick up fresh
wheels.

Releases are tagged `YYYY.MM.N` (e.g. `2026.05.1`); each tag has a GitHub
release with a table of pinned main-package versions. Envs are named
after the tag (`torch-2026.05.1`) so multiple versions live side-by-side.

## Install

```bash
make latest                   # build env from newest tag (auto-detects mac/linux)
conda activate torch-<tag>
```

Other targets:

```bash
make 2026.05.1                # build env from a specific tag
make list-tags                # show available tags
```

## Shell shortcuts

`shell/torch-env.sh` defines two functions; source it from your shell rc:

```bash
echo 'source /path/to/torch-env/shell/torch-env.sh' >> ~/.bashrc   # or ~/.zshrc
```

- `torch-latest` — activate the highest-version `torch-YYYY.MM.N` env
  installed locally. Repo-agnostic.
- `torch-repo` — from anywhere inside a consumer repo, activate the env
  pinned by `.torch-env-mac` / `.torch-env-linux`.

For devcontainers, add to your `.devcontainer/devcontainer.json`:

```jsonc
{
  "mounts": [
    "source=${localEnv:HOME}/path/to/torch-env,target=/opt/torch-env,type=bind,consistency=cached"
  ],
  "postCreateCommand": "echo 'source /etc/profile.d/conda.sh && source /opt/torch-env/shell/torch-env.sh' >> ~/.bashrc"
}
```

## Using torch-env from other repos

Each consumer repo (`zap-model`, `connectome-gnn`, ...) keeps its own
`pyproject.toml` declaring only repo-specific deps (not the shared base)
and a one-line `.torch-env-<plat>` file pinning a tag.

```bash
torch-repo                              # or conda activate torch-<tag>
cd path/to/repo-a && pip install -e .
cd path/to/repo-b && pip install -e .
```

Each `pip install -e .` writes an editable-install record into the env's
site-packages. Both repos are then importable from the same env.

**Conventions:**

- Distinct top-level import names per repo (`zap_model`, `connectome_gnn`)
  to avoid collisions.
- Loose dep pins in consumer `pyproject.toml` to avoid cross-repo conflicts.
- Don't re-list shared-env packages in consumer pyprojects.
- When a dep appears in 2+ consumer repos, promote it into this env's yaml
  and remove from the consumers.

Register one Jupyter kernel per env:

```bash
python -m ipykernel install --user --name torch-<tag>
```

## Layers

- **`environment.{linux,mac}.yaml`** — loose spec, edited by hand. No
  version pins. This is what the monthly upgrade job re-solves.
- **`lockfiles/requirements.{linux,mac}.txt`** — fully pinned `pip freeze`
  output, regenerated by CI on every release.

## Smoke test

`scripts/test_env.py` auto-detects the platform and checks that torch,
numpy, scipy import and that CUDA (linux) or MPS (mac) is available. Use
`--ci` to relax device checks to build-time checks (for CI runners without
a GPU).

## Monthly automation

`.github/workflows/monthly-upgrade.yml` runs on the 1st of every month
(and on manual dispatch):

1. Build both envs from their yamls on ubuntu-latest and macos-14.
2. Run `scripts/test_env.py --ci` on each.
3. Diff each new lockfile against the previous, filtered to packages
   named in the yaml's pip section ("main-list").
4. If any main-list package changed version on either platform, commit
   refreshed lockfiles, tag as the next `YYYY.MM.N`, and publish a
   GitHub release with a bumps summary and main-deps version table.

Transitive-only churn is ignored — no commit, no tag.
